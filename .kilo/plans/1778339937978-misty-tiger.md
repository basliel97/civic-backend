# Plan: Implement Proper Test Rescheduling Logic

## Overview
The user is confused about how test rescheduling should work. Currently, the `test_rescheduling` service simply creates a new test record, but "rescheduling" implies changing an existing scheduled test. We need to implement proper rescheduling logic that updates existing test records rather than creating duplicates.

## Current Codebase Analysis
- `test_rescheduling` fulfillment in `logic/fulfillment.ts` just inserts new test records
- `test_records` table has: user_id, test_type, scheduled_date, status, score, office_location, created_at, updated_at
- No eligibility restrictions currently (falls under default case)
- Current fulfillment: `INSERT INTO test_records (user_id, scheduled_date, status) VALUES ($1, $2, 'scheduled')`

## Problems with Current Implementation
1. **Creates duplicates**: Allows multiple scheduled tests for the same user
2. **No validation**: No check if user actually has a test to reschedule
3. **Poor naming**: "Rescheduling" when it actually schedules new tests
4. **Data integrity**: Multiple active scheduled tests don't make sense

## Implementation Options

### Option 1: True Rescheduling (Update Existing)
- Require user to have an existing scheduled test (status='scheduled', score=null)
- Update the scheduled_date of that existing record
- Update status to 'rescheduled' or keep as 'scheduled'
- Prevent rescheduling if no existing test
- **PROBLEM**: No service currently creates initial scheduled tests!

### Option 2: Initial Scheduling via Rescheduling
- Allow "rescheduling" to be used for initial test scheduling
- Rename service to "theory_test_scheduling" for clarity
- Remove "re" prefix since it's not rescheduling

### Option 3: Hybrid Approach (Recommended)
- If user has existing scheduled test: update it (true rescheduling)
- If user has no scheduled test: create new one (initial scheduling)
- Most flexible - handles both use cases

### Option 4: Create Separate Initial Scheduling Service
- Create new service for initial test scheduling
- Keep current rescheduling for updating existing tests
- Requires adding new service type and fulfillment logic

## Recommended Approach: Option 4 (Separate Services)

User has chosen Option 4: Create separate services for initial scheduling vs rescheduling.

## Implementation Steps

### 1. Create New Initial Test Scheduling Service
- **Database**: Add new bureau service with `automation_tag = 'theory_test_scheduling'`
- **Fulfillment**: Add new case `'theory_test_scheduling'` in `logic/fulfillment.ts`
  - Create new test record with status 'scheduled'
  - Set test_type from form response (default 'theory')
  - Handle office_location selection
- **Eligibility**: Add `'theory_test_scheduling'` case in `checkServiceEligibility`
  - Check if user already has a scheduled test (prevent duplicates)
  - Optional: Require driver's license
  - Optional: Age verification

### 2. Update Rescheduling Service Logic
- **Fulfillment**: Modify `test_rescheduling` in `logic/fulfillment.ts`
  - Query for existing scheduled tests (status='scheduled', score=null)
  - Allow user to select which test to reschedule (if multiple)
  - Update selected test's scheduled_date
  - Change status to 'rescheduled' or keep 'scheduled'
  - Handle case where no tests available to reschedule

### 3. Update Rescheduling Eligibility
- **Eligibility**: Add/modify `test_rescheduling` case in `checkServiceEligibility`
  - Check if user has at least one scheduled test
  - Optional: Prevent rescheduling within X days of test date
  - Optional: Limit total reschedule attempts
  - Optional: Require same test_type selection

### 4. Database Schema Considerations
- Current `test_records` table supports the logic
- May want to add `reschedule_count` field to track attempts
- Consider `test_type` filtering for rescheduling eligibility
- Status transitions: 'scheduled' → 'rescheduled' → 'scheduled' (for tracking)
