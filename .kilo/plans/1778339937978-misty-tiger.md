# Plan: Implement Duplicate Application Prevention

## Overview
The user discovered that current eligibility checks only prevent re-application based on RESULTS of previous applications (like having an active certificate), but don't prevent duplicate applications while previous applications are still PENDING admin review.

**Current Issue**: Users can apply multiple times for the same service before admin reviews the first application. For example:
- Apply for international verification 3 times → All 3 pending before admin sees any
- Apply for license replacement multiple times → Multiple pending replacements

**Required Solution**: Add a general duplicate prevention system that checks for any pending applications of the same service type, regardless of service-specific logic.

## Current Application Statuses
Applications can have these statuses:
- `pending_payment` (initial state)
- `paid`
- `under_review`
- `approved`
- `rejected`
- `printing`
- `ready_for_pickup`
- `dispatched`
- `completed`
- `cancelled`

### Implementation Change:
Instead of maintaining a selective service list, we'll apply pending prevention to ALL services by default, with an optional exclusion list for any services that genuinely need multiple pending applications.

## Implementation Approach

### Single Comprehensive Solution
Add a **universal pending application check** that runs at the beginning of `checkServiceEligibility` for specified services:

1. **Check BEFORE service-specific logic** runs
2. **Query for pending applications** of the same `automation_tag`
3. **Block if any found** with status NOT in `('approved', 'rejected', 'cancelled', 'completed')`
4. **Allow service-specific checks** to run after (for certificate/license status checks)

### Why This Approach?
- **Comprehensive**: Prevents duplicate pending applications for ALL specified services
- **Maintains existing logic**: Service-specific checks still work (international cert, taxi competency, etc.)
- **Consistent**: Same prevention logic across all services
- **Flexible**: Easy to add/remove services from prevention list

### 2. Create Pending Application Check Function

```typescript
async function checkPendingApplication(userId: string, automationTag: string): Promise<{hasPending: boolean, pendingApp?: any}>
```

### 3. Update checkServiceEligibility Function
**Add at the very beginning** of the function (before any service-specific logic):
```typescript
// Step 1: Universal pending application prevention (apply to ALL services except exclusions)
if (!PENDING_APPLICATION_EXCLUSIONS.includes(tag)) {
  const pendingCheck = await checkPendingApplication(userId, tag);
  if (pendingCheck.hasPending) {
    return {
      eligible: false,
      reason: 'pending_application_exists',
      message: `You have a pending application for this service (Application #${pendingCheck.pendingApp.id}). Please wait for it to be processed before applying again.`
    };
  }
}

// Step 2: Continue with existing service-specific eligibility checks
switch (tag) { ... }
```

### 4. Ensure Service-Specific Checks Still Run
The existing checks (international certificate validation, taxi competency, etc.) will still run AFTER the pending check passes, providing layered protection.

### 5. Status Definitions for Blocking
Block new applications if user has ANY previous application with status NOT in:
```typescript
const ALLOWED_REAPPLICATION_STATUSES = ['approved', 'rejected', 'cancelled', 'completed'];
```
This means pending statuses like `'pending_payment'`, `'paid'`, `'under_review'`, `'printing'`, `'ready_for_pickup'`, `'dispatched'` will block new applications.
